## Breaking changes

### Open browser tabs require a reload after upgrade

v0.5.5 removes the legacy base64 WebSocket attachment path that older browser tabs use. Any tab that was open before the upgrade will see a persistent "Protocol outdated" toast the next time it tries to attach a file. Reload the page to pick up the current version. No data is lost.

## Upgrade

Standard flow — no database migration, no `docker-compose.yml` change, and no env-var changes required:

```bash
cd /opt/pinchy
sed -i 's/PINCHY_VERSION=v0.5.4/PINCHY_VERSION=v0.5.5/' .env
docker compose pull && docker compose up -d
```

Existing agents are unaffected. Only new attachment uploads use the staging table added in this release.

## What's new

### Multipart attachment upload endpoint

Files are now uploaded via `POST /api/agents/<id>/uploads` as multipart form data before the message is sent. The server stages each file for up to 24 hours (status `staged`). When you send the message, staged files are promoted to a durable location (status `attached`) and included in the OpenClaw message as `attachmentIds`. Uploads that are never sent are cleaned up by an hourly garbage-collection sweep.

See [Attachments](/concepts/attachments/) for the full lifecycle.

### New audit-event family: `file.upload.*`

Three new audit events are emitted:

| Event type             | When it fires                                                 |
| ---------------------- | ------------------------------------------------------------- |
| `file.upload.staged`   | A file is accepted and staged after upload                    |
| `file.upload.attached` | A staged file is promoted to durable when its message is sent |
| `file.upload.expired`  | The GC sweep removes an orphaned staged upload                |

If you consume Pinchy audit events in a SIEM or alerting pipeline, update your allowlists to include these three event types.

### `sweepId` field in GC audit rows

Every `file.upload.expired` event emitted by a single GC sweep shares the same `sweepId` UUID in its `detail` payload. Use this field to correlate all expirations from one sweep in a single query — for example, to answer "what was cleaned up in last night's 02:00 run?"
